# testFilters

## Overview

`testFilters` evaluates how a specific flow is processed by filters (ACLs and firewall rules), reporting the permit/deny result and which line(s) matched.

**Location**: `projects/question/src/main/java/org/batfish/question/testfilters/`

**Question definition**: `questions/stable/testFilters.json`

## Why This Question Exists

When debugging ACL behavior or validating filter configurations, operators need to know:

1. **What happens to a specific packet?** Given a concrete flow, will the filter permit or deny it?
2. **Which line matched?** Understanding why a filter took a particular action requires knowing which rule matched.
3. **What was the evaluation path?** For filters that reference other filters (e.g., object-groups, nested ACLs), the full trace through all referenced structures is needed.

This question is the simplest filter analysis tool: given a flow and a filter, show what happens. Unlike `searchFilters` which finds flows matching criteria, or `filterLineReachability` which analyzes filter structure, `testFilters` answers the direct question "what does this filter do to this packet?"

The 2023 SIGCOMM paper (Section 4.4) discusses usability challenges in network verification, noting that "network engineers are used to concrete testing tools like traceroute and ping." `testFilters` provides this familiar concrete testing experience for ACL analysis — users specify a packet and see exactly what happens to it.

## How It Works

### High-Level Flow

1. **Resolve specifiers** - identify which nodes and filters to analyze based on `nodes` and `filters` parameters
2. **Generate test flows** - create concrete flows from `headers` constraints, inferring source IPs from `startLocation` when not specified
3. **Evaluate each flow against each filter** - run the flow through the filter's lines sequentially
4. **Generate traces** - record which lines were evaluated and matched
5. **Return results** - one row per (node, filter, flow) combination

### Flow Generation

The question generates concrete test flows from the provided header constraints:

1. If `srcIps` is specified, use those IPs directly
2. If `srcIps` is not specified but `startLocation` is, infer source IPs from the location's interface addresses
3. If neither is specified, use IP space from all interfaces on the target node

The flow generation uses BDDs (`BDDPacket`) to find a representative flow satisfying the constraints. The `FlowPreference.TESTFILTER` preference is used, which prefers flows with common port numbers and protocols.

### Filter Evaluation

For each (filter, flow) pair, evaluation proceeds via two parallel paths:

1. **Action determination**: `IpAccessList.filter()` returns a `FilterResult` containing the action (PERMIT/DENY) and the index of the matching line (if any).

2. **Trace generation**: `AclTracer.trace()` walks the filter structure and builds a `TraceTree` showing the evaluation path, including:
   - Which lines were examined
   - How nested ACL references were resolved
   - Which IP spaces or conditions matched

### Handling Nested Filters

When a filter line references another filter (via `PermittedByAcl`, `DeniedByAcl`, or `AclAclLine`), both the evaluator and tracer recurse into the referenced filter. The trace output shows this nesting structure, making it clear which sub-filter contributed to the final result.

### Trace Structure

The `Trace` column contains a list of `TraceTree` objects — hierarchical nodes representing the evaluation path:

```
TraceTree
├── traceElement: "Matched line 100 permit tcp any host 10.0.0.1 eq 443"
└── children: [
      TraceTree
      ├── traceElement: "Destination IP 10.0.0.1 permitted by '10.0.0.1/32'"
      └── children: []
    ]
```

**Key trace elements generated by `AclTracer`:**

| Expression Type | Trace Content |
|----------------|---------------|
| `MatchSourceIp` / `MatchDestinationIp` | IP address and which IP space matched |
| `MatchSourcePort` / `MatchDestinationPort` | Port number and matching range |
| `MatchIpProtocol` | Protocol number/name |
| `PermittedByAcl` / `DeniedByAcl` | Nested trace showing referenced ACL evaluation |
| `AndMatchExpr` / `OrMatchExpr` | Child traces for each sub-expression |
| `MatchHeaderSpace` | Sub-traces for each matched field (DSCP, TCP flags, etc.) |

**Trace filtering**: Only the path that led to the final action is included. Failed match attempts are discarded to keep traces concise.

**Negation handling**: For negated conditions like "except port 22" (`NotMatchExpr`), the inner operand's trace is always discarded — only the `NotMatchExpr`'s own trace element (if present) is preserved. This is a deliberate design choice noted in the code with a TODO: "how should we handle explaining 'did not match'?" Preserving sub-traces for expressions that returned false was deemed misleading. As a result, negated conditions may appear as a single trace element without details about what specifically didn't match.

### Zone-Based Firewalls and Composite Filters

Batfish converts vendor-specific zone-based firewall constructs (Cisco ASA security levels, Juniper security zones, etc.) into standard `IpAccessList` objects during configuration parsing. By the time `testFilters` processes them, they appear as regular ACLs.

This means:
- Zone-based policies are testable like any other filter
- The filter name may be auto-generated (e.g., `~ZONE_OUTGOING_ACL~zone-z3~`)
- Use the `filters` parameter with a regex to find zone-generated ACLs
- Composite filters (multiple filters combined) are similarly flattened

## Key Classes

| Class | Responsibility |
|-------|---------------|
| `TestFiltersAnswerer` | Orchestrates the analysis: resolves specifiers, generates flows, evaluates filters, builds result table |
| `TestFiltersQuestion` | Question parameters: nodes, filters, headers, startLocation |

| `FilterResult` | Simple container for action and matching line index |
| `IpAccessList.filter()` | Core filter evaluation logic (in `projects/common/`) |

## Output Schema

| Column | Description |
|--------|-------------|
| `Node` | The node where the filter exists |
| `Filter_Name` | Name of the evaluated filter |
| `Flow` | The concrete flow that was tested |
| `Action` | PERMIT or DENY result |
| `Line_Content` | Name or description of the matching line (or "no-match" if no line matched) |
| `Trace` | List of `TraceTree` objects showing the evaluation path |

The `Trace` column provides visibility into nested filter references. When a filter references other filters, the trace shows each level of evaluation.

## Parameters

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `nodes` | nodeSpec | No | Restrict analysis to filters on matching nodes (default: all nodes) |
| `filters` | filterSpec | No | Restrict analysis to matching filters (default: all filters) |
| `headers` | headerConstraint | Yes | Packet header constraints; `dstIps` is required |
| `startLocation` | locationSpec | No | Location to start tracing from; affects source IP inference |

## Performance Considerations

- **No data plane required**: `getDataPlane()` returns `false` - this question operates purely on the parsed configuration
- **Linear per filter**: Each filter is evaluated once per flow, with O(lines) complexity
- **Flow cross-product**: If multiple start locations match, a flow is generated for each, which can multiply the result set

## Known Limitations

1. **Single flow per constraint set**: The question generates one representative flow per (location, header constraint) combination. It does not exhaustively test all possible flows matching the constraints. Use `searchFilters` for exhaustive analysis.

2. **No stateful modeling**: Filters are evaluated statelessly. A line matching on TCP established flags is checked against the flow's flags, not against any session state. Real firewalls with stateful inspection may behave differently.

3. **Source IP inference**: When `srcIps` is not specified, source IPs are inferred from interface addresses. Nodes without interfaces produce flows with a dummy source IP.

4. **Line naming**: The `Line_Content` column shows the line's `name` field if set, otherwise "line:<index>". Filters without named lines may have less informative output.

## Common Sources of Confusion

### "Why does my ACL deny when I expected it to permit?"

The most common causes:

1. **Implicit deny**: Most ACLs have an implicit deny-all at the end. If no explicit line matches, the packet is denied.

2. **Wrong source IP**: If you didn't specify `srcIps`, the source IP is inferred from interface addresses. Check the `Flow` column to see what IP was actually tested.

3. **Earlier line matched**: A broader line earlier in the ACL may have matched first. Check the `Line_Content` column — it shows which line actually matched, not which line you expected.

4. **Protocol mismatch**: An ACL line like `permit tcp any any eq 80` won't match UDP traffic to port 80. Check that the protocol in your `headers` matches the ACL.

### "Why is the trace shallow or missing details?"

Traces only include elements with explicit `TraceElement` annotations in the code. Some conditions:

1. **Vendor variation**: Not all vendor parsers annotate all constructs with trace elements. Cisco IOS has the most complete tracing.

2. **Simple matches**: Very simple expressions (like `TrueExpr` or `FalseExpr`) may not generate traces since there's nothing meaningful to report.

3. **Failed paths discarded**: Traces only show the successful evaluation path. If a line didn't match, its trace is discarded.

### "Why did it pick this source IP / these ports?"

When header constraints allow multiple valid flows, `testFilters` picks one representative flow using `FlowPreference.TESTFILTER`. This preference:

- Biases toward common protocols (TCP, UDP, ICMP)
- Prefers well-known port numbers
- Uses heuristics for "interesting" traffic

If the chosen flow seems strange, specify more constraints in `headers` to narrow the selection.

### "Why didn't my flow match line N?"

This is a common and difficult question. `testFilters` shows which line *did* match, but not why other lines *didn't* match.

**Workarounds:**

1. **Use `searchFilters` with `matchLine`**: Ask "find me any flow that matches line N":
   ```python
   bf.q.searchFilters(filters="my-acl", action="matchLine 50").answer()
   ```
   If this returns a flow, compare it to your test flow to understand the difference. If it returns nothing, line 50 may be unreachable (use `filterLineReachability` to confirm).

2. **Manually check each condition**: Examine line N's match conditions and compare each against your flow's headers. The mismatch is often a single field (wrong protocol, port outside range, etc.).

3. **Test with modified headers**: Iteratively adjust your test flow's headers to isolate which condition is failing.

### "The filter behavior doesn't match the real device"

Batfish models L3/L4 packet headers but does not model:

- **L7 / application-layer inspection**: Deep packet inspection, URL filtering, application identification
- **L2 headers**: MAC addresses, VLAN tags (except where explicitly modeled)
- **Nested packets**: Packets inside tunnels, GRE payloads, IPsec ESP contents
- **Dynamic state**: Connection tracking, session tables, rate limits

When these features are configured, Batfish makes pragmatic default assumptions that may not match real device behavior. This is particularly common with **Palo Alto Networks** firewalls, where application-aware policies and threat prevention features are not modeled.

**Best practice**: Use `testFilters` for L3/L4 ACL logic validation. For L7 or stateful policies, treat Batfish results as a lower bound — real devices may deny additional traffic based on unmodeled features.

### "How do I test zone-based firewall policies?"

Zone-generated ACLs have auto-generated names like `~ZONE_OUTGOING_ACL~zone-name~`. To find them:

```python
# List all zone-related filters
bf.q.testFilters(
    filters="/~ZONE/",  # regex matching zone ACLs
    headers=HeaderConstraints(dstIps="10.0.0.1", applications=["http"])
).answer()
```

## Related Questions

- **`searchFilters`**: Find flows that match or don't match a filter (uses BDDs for exhaustive search)
- **`filterLineReachability`**: Find unreachable lines in filters (structural analysis)
- **`compareFilters`**: Compare filter behavior between two snapshots (differential analysis)
- **`findMatchingFilterLines`**: Find which lines in a filter match a given header space

### When to Use Each

| Goal | Question |
|------|----------|
| "What happens to this specific packet?" | `testFilters` |
| "Is there any packet that gets denied?" | `searchFilters` |
| "Are there dead rules in my ACL?" | `filterLineReachability` |
| "Did my ACL change break anything?" | `compareFilters` |
| "Which rules match traffic to 10.0.0.0/8?" | `findMatchingFilterLines` |

## Design Evolution

The question has evolved significantly, particularly around trace generation:

- **batfish/batfish#5425**: `AclTracer` shifted from flat string traces to hierarchical `TraceTree` structures
- **batfish/batfish#5469**: `testFilters` updated to return `TraceTree` objects
- **batfish/batfish#5475**: Optimization to only trace lines with explicit `TraceElement` annotations
- **batfish/batfish#5539**: Unified packet header constraint conversion
- **batfish/batfish#5543**: Performance improvements for flow and trace generation
- **batfish/batfish#7883**: Location-independent IP space specifier enhancements
